**Crate/repo ownership:**

**Mission:** Build everything *around* the compiler so the language is usable in editors, has documentation, and has more example programs. None of this touches compiler internals.

## Why this allocation

- **Tree-sitter is a declarative grammar.** `grammar.js` is essentially a BNF file. No Rust cleverness required.
- **LSP is message routing.** `tower-lsp` handles the JSON-RPC plumbing; you write handlers that call into the compiler's public API.
- **Docs are docs.** The 14 example programs already exist; tour them.
- **More example programs** is a self-contained task. Write `.pp` files, run them through `pipe-lang run`, capture expected output.

All four pieces are **completely independent** of the other team members' work. You can ship everything in parallel with the compiler being built. The LSP server is the only piece that depends on a stable compiler API; that lands on Day 12.

**Grammar (Node.js style):**

```javascript
module.exports = grammar({
  name: "pipe_lang",

  extras: $ => [/\s/, $.line_comment, $.block_comment],

  rules: {
    source_file: $ => repeat($._top_level),

    _top_level: $ => choice(
      $.let_def,
      $.type_def,
      $.use_def,
    ),

    let_def: $ => seq(
      "let",
      $.identifier,
      optional(seq(":", $._type_expr)),
      "=",
      $._expr,
    ),

    type_def: $ => seq(
      "type",
      $.identifier,
      optional($.type_params),
      "=",
      $._type_expr,
    ),

    use_def: $ => seq(
      "use",
      $.module_path,
      optional(choice(
        seq("::", "*"),
        seq("::", "{", commaSep($.identifier), "}"),
      )),
    ),

    module_path: $ => sep1($.identifier, "::"),

    type_params: $ => seq("<", commaSep($.identifier), ">"),

    // Type expressions
    _type_expr: $ => choice(
      "()",
      $.primitive_type,
      $.type_identifier,
      $.generic_type,
      $.function_type,
      $.tuple_type,
      $.record_type,
    ),

    primitive_type: $ => choice(
      "i8", "i16", "i32", "i64",
      "u8", "u16", "u32", "u64", "usize",
      "f32", "f64",
      "bool", "str",
    ),

    generic_type: $ => seq($.type_identifier, "<", commaSep($._type_expr), ">"),

    function_type: $ => seq(
      optional(seq("(", commaSep($._type_expr), ")")),
      "->",
      $._type_expr,
    ),

    // Expressions
    _expr: $ => choice(
      $.identifier,
      $.int_literal,
      $.float_literal,
      $.string_literal,
      $.template_literal,
      $.bool_literal,
      $.unit_literal,
      $.lambda,
      $.application,
      $.binary_op,
      $.unary_op,
      $.if_expr,
      $.match_expr,
      $.do_block,
      $.block,
      $.record_literal,
      $.tuple_literal,
      $.array_literal,
      $.field_access,
      $.method_call,
      $.parenthesized,
    ),

    template_literal: $ => seq(
      "`",
      repeat(choice($.template_str_part, $.template_hole)),
      "`",
    ),

    template_hole: $ => seq("${", $._expr, "}"),

    lambda: $ => seq(
      "(",
      commaSep($.lambda_param),
      ")",
      "=>",
      $._expr,
    ),

    if_expr: $ => seq(
      "if",
      $._expr,
      "then",
      $._expr,
      "else",
      $._expr,
    ),

    match_expr: $ => seq(
      "match",
      $._expr,
      "{",
      repeat(seq($._pattern, "=>", $._expr, ",")),
      "}",
    ),

    do_block: $ => seq(
      "do",
      "{",
      repeat(choice(
        seq($.identifier, "<-", $._expr, ";"),
        seq($._expr, ";"),
      )),
      optional($._expr),
      "}",
    ),

    // Patterns
    _pattern: $ => choice(
      "_",
      $.identifier,
      $.int_literal,
      $.string_literal,
      $.constructor_pattern,
      $.tuple_pattern,
      $.record_pattern,
      $.list_pattern,
    ),

    constructor_pattern: $ => prec(1, seq($.identifier, optional(seq("(", commaSep($._pattern), ")")))),
    list_pattern: $ => choice("[]", seq("[", $._pattern, "]"), seq($._pattern, ":", $._pattern)),

    // ... etc.
  },
});
```

## Deliverable B: LSP Server (Days 4–7)

**Crate:** `crates/pipe-lang-lsp/` (new)

**Dependencies:** `tower-lsp`, `serde_json`, plus the compiler's public APIs (lex, parse, typecheck, IR — all from `dijith.md`)

**Files:**
- `src/main.rs` — binary entry, `LspService::new(Backend::new(), socket)`
- `src/backend.rs` — `LanguageServer` impl
- `src/handlers.rs` — `did_open`, `did_change`, `hover`, `completion`
- `src/diagnostics.rs` — convert `CompilerError` → `lsp_types::Diagnostic`

**0.1 scope (per the spec):**
- `textDocument/didOpen` → run lex + parse + typecheck, send diagnostics
- `textDocument/didChange` → re-run pipeline on change, send updated diagnostics
- `textDocument/hover` → return the inferred type at the position (uses `TypedProgram::type_for_span`)
- `textDocument/publishDiagnostics` — the diagnostics stream

**0.2 scope (NOT in 0.1):**
- `textDocument/definition` — go-to-definition
- `textDocument/completion` — autocomplete
- `textDocument/rename` — symbol rename
- `textDocument/formatting` — formatter (use `rust-analyzer`-style heuristics)

**API:**
```rust
struct Backend {
    client: Client,
    compiler: Arc<CompilerSession>,  // wraps lexer/parser/typechecker
}

#[tower_lsp::async_trait]
impl LanguageServer for Backend {
    async fn initialize(&self, _: InitializeParams) -> Result<InitializeResult> { ... }
    async fn did_open(&self, params: DidOpenTextDocumentParams) { /* run pipeline, send diagnostics */ }
    async fn did_change(&self, params: DidChangeTextDocumentParams) { /* re-run */ }
    async fn hover(&self, params: HoverParams) -> Result<Option<Hover>> { /* type_for_span */ }
}
```

**Test suite (5 tests):**
- `lsp_initializes_with_capabilities` — initialize handshake returns correct capabilities
- `lsp_reports_diagnostics_on_open` — feed a broken file, assert a `Diagnostic` is sent
- `lsp_hover_returns_type` — feed a file with `let x = 5`, hover over `x`, assert `i32`
- `lsp_did_change_re_runs` — change a file, assert new diagnostics flow
- `lsp_handles_template_literal` — hover over a `${n}` hole returns the type of `n`

## Common pitfalls

1. **Tree-sitter can't handle your grammar on the first try.** Use `tree-sitter generate` and `tree-sitter test` iteratively. The `corpus/*.txt` files show expected trees.
2. **LSP `did_change` is incremental.** Don't re-tokenize the whole file; use the versioned changes. For 0.1, re-tokenize is fine (slow but correct).
3. **Hover must return `Some(Hover)` for in-range positions only.** Off-range or whitespace positions return `None`.
4. **Examples must compile.** Run `pipe-lang run <file>.pp` before committing. If dijith's compiler is not ready, validate the syntax against `pipe-lang.md` by hand.
5. **Don't write clever grammars.** Tree-sitter grammars are notoriously hard to debug; stick close to the spec.
6. **The LSP only depends on dijith's public API.** Don't import from `lexer/src/private.rs` — use the public `parse()` / `infer_program()` functions.
